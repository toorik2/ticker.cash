Docs · BCH/USD ticker
v12.1 · chipnet

Decentralized BCH/USD ticker.

A 7-notary federation co-witnesses prices from 13 operator-diverse exchanges; any publisher relays them into a covenant on Bitcoin Cash that commits the per-cycle median on chain. No admin keys; the covenant is the only on-chain rule.

§01Overview

Ticker is a price feed that lives entirely on Bitcoin Cash chipnet. Every cycle (~60 s when funded) it emits a new on-chain commitment containing the median BCH/USD price witnessed during that cycle, signed by at least 7 distinct publishers. Consumers read the price either by spending a Ticker NFT in a tx (atomic on-chain) or by polling the JSON API.

There is no operator and no upgrade path after genesis (genesis itself is operator-orchestrated, one-shot). The Oracle covenant address is permanent for the life of the deploy; behavioural changes require minting a new covenant at a new address. The notary federation is the only off-chain trust assumption, and the design constrains what it can do.

The on-chain code, in full, is three CashScript covenants: Oracle, PublisherSlot, and Ticker. There are no proxies, no admin paths, no off-chain governance signers.

§02Architecture

Five roles, all replaceable:

RoleWhat it isHow many
NotaryStateless HTTP daemon holding one Schnorr key from the federation OR-list. Fetches the price from its assigned CEX over TLS and signs (serverName, sourceId, price, ts, cycleSeq).7 (federated, independent operators)
PublisherDaemon that requests notary signatures, refreshes its persistent PublisherSlot NFT via slot.attest(), then races to broadcast Oracle.update consuming peer slot inputs. No special role; anyone with chipnet funds can run one.13 (one per source; permissionless)
PublisherSlotPersistent mutable NFT, one per (publisher, sourceId) pair. Each cycle the publisher calls slot.attest(), which verifies the notary sig + CN/SAN hash and rewrites the slot commitment (preserves (category, sourceId, publisherPkh) identity; new outpoint at every cycle; new (price, timestamp, cycleSeq)).13 (one per source; genesis-minted, category-closed)

Cycle flow

  1. Publisher polls the Oracle UTXO to learn the current prevTs and cycleSeq.
  2. Publisher requests a signature from one of the 7 notaries (round-robin) for its assigned source.
  3. Publisher builds + broadcasts a PublisherSlot.attest tx. The slot covenant verifies the notary sig, the CN/SAN hash, the publisher sig, and the monotonic cycleSeq advance, then rewrites the slot commitment in place. Output: the same PublisherSlot NFT with a refreshed commitment (same category, same identity, new outpoint).
  4. Publisher waits up to ~25 s for peer slot refreshes from other publishers to appear in mempool.
  5. Once ≥7 distinct slot refreshes are seen for this cycleSeq, publisher builds Oracle.update consuming the Oracle UTXO + ≥7 PublisherSlot inputs, re-emitting each slot unchanged at the matching output index, with the median price as the new commitment.
  6. First publisher to broadcast wins the cycle. Losing publishers' txs fail with txn-mempool-conflict and the cycle advances; they retry on the new tip.

Chain-state, not consensus

There is no central coordinator. Each publisher acts on what it sees in the chain + mempool. The Oracle UTXO is the single source of truth for cycle state; the rest is replication and racing.

§03Trust model

The trust root is the 7-key notary federation. Each notary independently fetches the price from its pinned CEX over TLS and signs (serverName, sourceId, price, ts, cycleSeq) with its Schnorr key. The Oracle covenant trusts these signatures and nothing else off-chain.

Publishers are interchangeable relays. They cannot fake notary sigs. They cannot alter the price. They cannot skip the quorum check. The only adversarial power a publisher has is to broadcast a different median if they assemble a different subset of slot inputs, but the covenant enforces position-checked median across N ≥ 7 distinct publishers, so a single publisher cannot bias the result.

Sources are operator-diverse. 13 endpoints: 9 USD-quoted spot markets (Kraken, Coinbase, Gemini, Binance.US, Bitstamp, Crypto.com, Bitfinex, EXMO, Independent Reserve — 4 US, 5 non-US), 2 USDC (OKX, KuCoin), 2 USDT (Bybit, HTX). No two share an operator family; no quote-currency dominates. A position-checked median absorbs ≤6 outliers without bias — both stablecoin clusters could depeg the same direction simultaneously and the median still lands in the 9-USD cluster.

The covenant is the only on-chain rule. It enforces:

  • ≥ 7 distinct publishers attest per cycle (the quorum ratchets up with active committee at ~50%, decays 10%/cycle if participation drops)
  • Notary-attested time is strictly forward; replay protection comes from monotonic cycleSeq enforced inside PublisherSlot.attest() — a stale attestation cannot be replayed because the slot's cycleSeq has already advanced past it. The minimum-stride floor (≥ 30 s) is enforced by BOTH the covenant (Oracle.cash: require(newTs - prevTs >= 30)) AND the publisher daemon (scripts/publisher.ts SLOT_WAIT_MS=25s + stride-floor wait).
  • Median is position-checked across the sorted slot prices, not computed off-chain and trusted
  • Anti-Sybil rests on two complementary, load-bearing checks: (a) the slot category is closed forever at genesis by CashTokens consensus, so the publisher set is fixed and cannot be inflated by minting new slots; (b) within each Oracle.update tx, the covenant enforces distinct, monotonically-sorted publisherPkh across slot inputs, so a publisher cannot inflate quorum by submitting their own slot twice.
  • Slot inputs are pinned to the v12 slot category — no synthetic slot inputs can be substituted.
  • The 2 Mutable Ticker NFTs minted per cycle carry the same commitment shape; consumer covenants can pin the Oracle category to refuse impostor Tickers

No admin keys. No upgrade path. The Oracle UTXO chain is self-perpetuating. The only way to update behaviour is to deploy a new covenant at a new address — which observers will see as a fresh address with no history.

Federation collusion model

What a colluding majority (≥ 4 of 7) can do:

  • Sign attestations for arbitrary prices the publishers will then propagate

What a colluding majority cannot do:

  • Rewind the cycleSeq counter — strictly monotonic in the covenant
  • Replay a stale notary signature — PublisherSlot.attest() requires newCycleSeq > slot.cycleSeq (strict-greater, not ≥), so once a slot has advanced past a cycle, that cycle's signature is dead
  • Skip the ≥ 7 distinct-publisher quorum check — enforced in script
  • Hide a cycle from observers — every state transition is an on-chain tx
  • Run the chain faster than the 30 s minimum stride enforced by Oracle.cash (require(newTs - prevTs >= 30)); typical cadence is ~60 s when funded

A 7-key M-of-N federation with operator-diverse keys — bribing 4 keys held by independent operators is the new collusion threshold. Adding an 8th notary requires deploying a fresh PublisherSlot covenant + migrating the slot fleet (a full hard cutover at genesis); the new address coexists alongside the old.

§04Signatures

Six signatures appear on the path from "an exchange quoted a price" to "your dApp's tx settles." Three are produced and verified per slot refresh — the heart of the trust model. The other three are standard BCH input sigs. No single operator holds a master key; every signature ties to a specific publisher, source, and cycle, so misuse is structurally detectable.

Step 1 · Slot refresh (×13 publishers in parallel, ~60 s cadence)

Each publisher fetches a price from its assigned exchange over TLS and asks one of the 7 notaries to co-witness it. Two off-chain Schnorr sigs feed one BCH tx that rewrites the publisher's persistent slot NFT.

Step 2 · Oracle.update (any publisher races, once ≥ 7 slots are refreshed)

The race winner builds a tx consuming the Oracle UTXO + the ≥ 7 freshest slot UTXOs, re-emits each slot unchanged at its matching output index, computes the median, and mints two Ticker NFTs. No off-chain sig — the covenant is the only judge of validity; the race winner's wallet just pays the tx fee.

Step 3 · Consumer (any time the Tickers exist on chain)

A consumer dApp builds its own tx with a Ticker NFT at input[1] and the consumer's own covenant at input[0]. The Ticker covenant enforces identical re-emit at output[1]; the consumer covenant enforces whatever price gate it wants (e.g., release iff medianPrice >= strike). The act of spending the Ticker makes the consumer tx a chain-descendant of the Oracle.update that minted it — co-finality is structural, not advisory.

Why each binding matters

  • publisherPkh inside the notary sig. A notary cannot sign one bundle and have ten publishers replay it. Each notary sig is for one publisher in one cycle. A publisher who tried to use someone else's notary sig would fail the slot's identity check (the slot's pinned publisherPkh would not match the notary's signed message).
  • hash160(serverName) inside the publisher sig. A publisher cannot ask the Coinbase-side notary for a sig and quietly relabel the price as Kraken's. The covenant cross-checks that hash160(serverName) matches the 20-byte CN/SAN hash pinned at the slot's sourceId slot.
  • cycleSeq inside both sigs. A sig produced for cycle N cannot be replayed at N+1: the covenant requires newCycleSeq > oldCycleSeq (strict-greater), and the cycleSeq appears inside the signed bytes, so altering it would break the sig.
  • No off-chain sig on Oracle.update. Anyone can broadcast the update tx. The covenant is the only judge of validity. The race-winner's wallet sig (sig 4) just pays the tx fee — they don't gain any special authority by being first.
  • Consumer covenant's own gate. The Ticker covenant only enforces structural re-emit; it does not enforce a price-meaning constraint. Each consumer pins its own logic — strike price, freshness window, recipient pubkey — into the covenant that holds its locked BCH (see § 05 for the reference example).
What no signature in this system grants. No signer can change the rules of the covenant, can roll back a confirmed cycle, can override the median computation, or can mint new slots after genesis. Every signature is a witness inside a fixed game; the rules of the game are in contracts/Oracle.cash + contracts/PublisherSlot.cash, deployed once, immutable forever.

§05Reading the price

Two paths: on-chain atomic (recommended for dApps; spend a Ticker NFT as a tx input, read its commit) or JSON API (recommended for off-chain UIs; just GET /api/v1/price).

On-chain: spend a Ticker

Each cycle mints 2 Ticker NFTs at a fixed address (see § 08 Live contracts). They are mutable (capability 0x01), anyone-can-spend, and self-replicate via the Ticker covenant. A consumer's tx that spends a Ticker becomes a chain-descendant of the Oracle.update that produced it — BCH consensus refuses to confirm the consumer's tx in any chain that excludes the Oracle.update. That is co-finality, atomic by construction.

Minimal tx shape:

  • input[0] — your consumer contract
  • input[1] — Ticker NFT from the pinned Oracle category
  • output[0] — your business logic (payment, mint, settle…)
  • output[1] — identical Ticker re-emit (Ticker covenant enforces this, mutable cap, same commit)
  • output[2] — optional change

Read price and lastTs from tx.inputs[1].nftCommitment — the Ticker's 17-byte 0x80 commit, distinct from the Oracle UTXO's 19-byte 0x60 state commit. The 17-byte layout is:

0x80 | seq(4) | lastTs(4) | medianPrice(8)

All multi-byte fields are little-endian. medianPrice / 10^8 = USD. Pin the Oracle category as a constructor arg (LE-reversed; see § 08) so an attacker cannot feed your contract a Ticker from a different oracle.

A working reference covenant — PriceGatedRelease (locks BCH, releases on price ≥ strike) — lives at contracts/examples/PriceGatedRelease.cash in the source tree. ~64-byte script body; ~145 bytes including the four constructor args (pubkey + strike + oracle category + minLastTs). Drop your own freshness gate on lastTs; avoid tx.locktime, that introduces a chain-MTP lag the design was built to escape.

Off-chain: JSON API

$ curl -s https://usd.ticker.cash/api/v1/price

{
  "medianUsd": 344.94,
  "scale": "usd-e8",
  "scaledValue": "34494000000",
  "lastLocktime": 1779818724,
  "seq": 11,
  "activeCount": 7,
  "network": "chipnet",
  "deployedAt": 1779210000,
  "fetchedAt": 1779818790,
  "ageSec": 66,
  "stub": false
}

CORS is open; auth is none. stub: true indicates the price should not be trusted as live. Two branches: (a) the backend couldn't read the Oracle UTXO from chain — medianUsd will be null and the HTTP status is 503; (b) the latest cycle is ≥ 5 minutes old — medianUsd is real but stale, ageSec ≥ 300, HTTP 200. Always check stub AND consult /api/v1/health for component-level diagnosis.

§06Operating a node

There are two kinds of node, both stateless and small. The unified binary scripts/ticker-node.ts wraps either or both.

Notary

A stateless HTTP daemon that holds one Schnorr key from the federation's 7-key OR-list. Exposes POST /sign; on each request fetches the current price from its assigned CEX, signs (serverName, sourceId, price, ts, cycleSeq), returns the signature. The federation's seven keys live independently — separate hosts, separate operators. Adding an 8th notary requires deploying a fresh PublisherSlot covenant + migrating the slot fleet (a full hard cutover at genesis).

$ ticker-node --notary --slot 0   # listens on 127.0.0.1:8081

Publisher

A daemon that polls the Oracle UTXO, requests notary signatures for its assigned source, refreshes its persistent PublisherSlot NFT via slot.attest(), waits ~25 s for peer slot refreshes from other publishers, then builds and broadcasts the Oracle.update tx consuming ≥7 slot inputs and re-emitting each one unchanged at the matching output index. First publisher to broadcast wins the cycle; losers retry on the new tip.

A publisher needs ~12 k sats per cycle (slot refresh + Oracle.update share of fee + Ticker dust). No registration, no allowlist beyond holding the genesis-minted PublisherSlot NFT for the assigned source.

$ ticker-node --publisher --slot 0 \
  --notary-url https://notary-0.ticker.cash \
  --notary-url https://notary-1.ticker.cash \
  --notary-url https://notary-2.ticker.cash \
  --notary-url https://notary-3.ticker.cash \
  --notary-url https://notary-4.ticker.cash \
  --notary-url https://notary-5.ticker.cash \
  --notary-url https://notary-6.ticker.cash

Bundled deploy

Federation operators typically bundle both daemons for their slot (no HTTP roundtrip for own attestations); community operators run publisher-only.

$ ticker-node --notary --publisher --slot 0
Source code. Lives on github.com/toorik2/ticker.cash.

§08Live contracts

Current chipnet deployment. Live addresses and the most recent commitment are on the home page sidebar.

  • Oracle — singleton covenant; consumes ≥ 7 PublisherSlot inputs per cycle, re-emits each one unchanged at the matching output index, emits new state commitment + 2 Ticker NFTs.
  • PublisherSlot — persistent mutable NFT, one per (publisher, sourceId) pair. Minted ONCE at genesis; category closed forever by CashTokens consensus. Each cycle, slot.attest() verifies the notary sig + CN/SAN hash and rewrites the slot commitment in place.
  • Ticker — mutable, self-replicating NFT carrying the latest 17-byte commitment. Consumers spend a Ticker to read the price atomically.

Source for all three covenants is at contracts/ in the repo. Compiled artifacts are at contracts/artifacts/.

§09JSON API

GET /api/v1/price — no auth, CORS open, returns the latest known Oracle state.

{
  "medianUsd": 344.94,         // float USD (medianPrice / 1e8); null if stub branch (a)
  "scale": "usd-e8",           // u64 LE scale; multiply by 1e-8 for USD
  "scaledValue": "34494000000",// raw u64 from on-chain commit
  "lastLocktime": 1779818724,  // notary-attested ts of last cycle (unix seconds)
  "seq": 11,                   // monotonic cycle counter from Oracle commit
  "activeCount": 7,            // distinct publishers in last cycle (also the ratchet input)
  "network": "chipnet",        // chipnet | mainnet
  "deployedAt": 1779210000,    // unix seconds; genesis-tx mined-at for this deploy
  "fetchedAt": 1779818790,     // unix seconds; backend's wall-clock when this response was built
  "ageSec": 66,                // fetchedAt - lastLocktime; ≥ 300 implies stub: true (branch b)
  "stub": false                // see explanation below
}

stub: true has two branches: (a) the backend couldn't read the Oracle UTXO — medianUsd is null, HTTP 503; (b) the latest cycle is ≥ 5 minutes old — medianUsd is real but stale, ageSec ≥ 300, HTTP 200. Useful for status pages, alerting, and dashboards. For dApp consumption, prefer the on-chain Ticker path described in § 05.

GET /api/v1/health

Returns:

{
  "healthy": true,
  "stalenessThresholdSec": 300,
  "node":     { "ok": true, "blockHeight": 224801 },
  "fulcrum":  { "ok": true, "tipHeight":    224801 },
  "lastCycle": {
    "seq": 11,
    "locktime": 1779818724,
    "ageSec": 66,
    "activeCount": 7

  "deployedAt": 1779210000,

healthy is true iff node + fulcrum are reachable AND lastCycle.ageSec < stalenessThresholdSec. Use this endpoint for status-page integrations — it surfaces component-level state that /api/v1/price collapses into a single stub bit.
